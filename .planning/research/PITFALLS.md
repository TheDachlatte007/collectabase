# Pitfalls Research

**Domain:** Multi-type collectible tracker (games, figures, Funko Pops, vinyl, comics)
**Researched:** 2026-03-17
**Confidence:** HIGH — based on direct codebase inspection plus known-encountered issues documented in PROJECT.md

---

## Critical Pitfalls

### Pitfall 1: Storing External Image URLs Instead of Caching Them at Save Time

**What goes wrong:**
`cover_url` is saved with the raw external URL returned by HobbyDB/MFC/eBay/GameTDB lookup. The URL appears to work at save time, then silently breaks hours or days later. The user sees broken images in their collection view — this is the issue already present in the live app.

**Why it happens:**
The metadata lookup returns the URL and the frontend sends it directly in the `GameCreate` payload. The `cache_remote_cover()` helper exists but is only called in the enrich/cover-placeholder endpoints, not in the main save path (`POST /api/games`). Adding a new item type (anime figure, Funko Pop) that relies on eBay image URLs makes this worse because eBay CDN URLs can have very short TTLs or require authentication to resolve.

**How to avoid:**
Call `cache_remote_cover(game.cover_url)` inside `create_game()` and `update_game()` in `games.py` before committing to the database. Gate on `startswith(("http://", "https://"))` — same guard already used in the helper itself. This guarantees every stored URL is a local `/uploads/...` path regardless of which lookup source provided it.

**Warning signs:**
- `cover_url` values in the DB that start with `https://` for item types other than games
- `cover_url` values containing `hobbydb.com`, `myfigurecollection.net`, `ebayimg.com`, or CDN subdomains
- Any `cover_url` starting with `//` (protocol-relative) — IGDB returns these and `cache_remote_cover()` already guards against them but they will display as broken without the `https:` prefix fix

**Phase to address:**
Image caching milestone (the current active milestone). This is the primary deliverable — wire `cache_remote_cover()` into the save path for all item types.

### Pitfall 2: Migrating Columns That Are Already Present on Live Databases

**What goes wrong:**
A migration calls `op.add_column()` without checking whether the column exists. If the column was created outside Alembic (e.g., during development, by hand, or by a previous migration that ran partially), Alembic raises `OperationalError: duplicate column name` and halts. Every subsequent `alembic upgrade` then fails until the database is manually patched.

**Why it happens:**
This has already happened: `b7c8d9e0f1a2_add_collectible_metadata_columns.py` added `_column_exists()` guards precisely because the columns (`comicvine_id`, `hobbydb_id`, `mfc_id`) were already present in some databases. The same failure mode will recur each time a new collectible type requires new columns.

**How to avoid:**
Every migration that adds columns to an existing table must use the `_column_exists()` guard pattern already established in `b7c8d9e0f1a2`. Do not rely on Alembic's `batch_alter_table` autogenerate to handle this — it will not. New migrations must follow the pattern:
```python
def upgrade() -> None:
    if not _table_exists("games"):
        return
    for col in ["new_col_a", "new_col_b"]:
        if not _column_exists("games", col):
            op.add_column("games", sa.Column(col, sa.String(), nullable=True))
```

**Warning signs:**
- Any new migration that calls `op.add_column()` without a preceding `_column_exists()` check
- Autogenerated migrations from `alembic revision --autogenerate` — these never include existence guards and should always be reviewed before running

**Phase to address:**
Any phase that extends the schema (adding type-specific metadata columns). The pattern from `b7c8d9e0f1a2` is the template — copy it, do not deviate.

### Pitfall 3: SQLAlchemy Model and Alembic Migration Diverging (Schema Drift)

**What goes wrong:**
A column is added to `models.py` (`Game` class) without a corresponding Alembic migration. On a fresh database the column exists (created by `Base.metadata.create_all`). On an existing production database the column is missing, causing `OperationalError: table games has no column named X` on save. The app silently works for the developer but fails in production.

**Why it happens:**
This is already documented as an experienced issue in the PROJECT.md context. SQLAlchemy's `Base.metadata.create_all` and Alembic migrations are completely independent. Adding a field to `Game` does not create a migration. The developer tests on a freshly created dev DB where `create_all` ran and never notices.

**How to avoid:**
The constraint stated in PROJECT.md is the correct policy: "Only add migrations for fields actually used by the live save path." Operationally: after every change to `models.py`, immediately write and test the corresponding migration. Use the `_column_exists()` guard. Never rely on `create_all` for production schema changes.

**Warning signs:**
- A column in `models.py` that has no matching `op.add_column()` call in any migration file
- `alembic current` reporting `head` but the DB is missing a column visible in `models.py`
- `INSERT` or `UPDATE` succeeding on dev but failing on a long-lived DB copy

**Phase to address:**
Every schema-touching phase. Enforce as a phase exit criterion: run `alembic upgrade head` against a copy of the production DB and verify no errors before merging.

### Pitfall 4: The `hobbydb` and `mfc` Sources Are eBay Masquerading Under Different Names

**What goes wrong:**
`lookup_hobbydb_title()` and `lookup_mfc_title()` both proxy to `_lookup_ebay_search_title()` and rename the source field. The frontend displays these as "HobbyDB" and "MFC" but the data — especially image URLs — comes from eBay's Browse API. eBay image URLs (`i.ebayimg.com`) are temporary, listing-specific, and can disappear when a listing ends. Caching them (see Pitfall 1) is therefore mandatory, not optional, for these sources.

**Why it happens:**
The original HobbyDB and MFC APIs had hotlink protection and were unreliable, so eBay was used as a fallback. The source masking (`r["source"] = "hobbydb"`) means the frontend has no idea it is displaying eBay data. There is also no fallback if the eBay token is missing or expired.

**How to avoid:**
- Always run `cache_remote_cover()` on any URL returned by these sources before saving
- Document internally (in code comments) that "hobbydb" and "mfc" sources are eBay-backed
- Ensure eBay credentials are validated at startup and that missing credentials produce a clear user-facing error, not a silent empty result
- Do not use eBay image URLs as "permanent" metadata: they are auction snapshots

**Warning signs:**
- `cover_url` values containing `i.ebayimg.com` in the database
- Empty results from hobbydb/mfc lookup with no error shown to the user (this happens when `get_ebay_token()` returns `None`)
- eBay token errors showing up in server logs without surfacing in the UI

**Phase to address:**
Image caching milestone (current). The token error path should also be surfaced in the UI as part of this work.

### Pitfall 5: `platform_id` Required Logic Breaks for Non-Game Item Types

**What goes wrong:**
The form in `AddGame.vue` marks the platform field as required only when `item_type === 'game'`. However, `POST /api/games` accepts `platform_id: null` for any type, and the DB schema has no NOT NULL constraint on `platform_id`. This is correct behavior. The pitfall is the inverse: if a future migration or schema change adds a NOT NULL constraint to `platform_id`, all non-game items (figures, comics, vinyl) would fail to save.

There is also a subtler issue: the stats query groups by `p.name` (platform name) using a `LEFT JOIN`. Items without a platform aggregate into a "No Platform" bucket. If new item types (figures, Funko Pops) are added and they all have no platform, the stats page "by_platform" breakdown becomes polluted with a large "No Platform" bucket that mixes everything non-game together.

**Why it happens:**
The schema and API were designed for games where platform is meaningful. Non-game types have no natural platform concept but are stored in the same table. The stats queries do not filter by `item_type` when breaking down by platform.

**How to avoid:**
- Keep `platform_id` nullable at the DB level — never add a NOT NULL constraint
- Update stats queries to filter or separate non-game items when displaying "by platform" breakdowns
- Consider a "No Platform" display name that is item-type aware (e.g., show "No Platform" only for types where platform is expected)

**Warning signs:**
- Stats page showing a large "No Platform" entry that represents multiple different item types mixed together
- Any migration that touches `platform_id` nullable setting

**Phase to address:**
Stats cleanup milestone (deferred per PROJECT.md). For the current milestone, ensure no code changes accidentally tighten the nullable constraint.

## Technical Debt Patterns

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| Masking eBay as "hobbydb"/"mfc" source | Existing frontend source-label logic works without changes | When eBay changes its API or the real HobbyDB/MFC become available, the masking creates confusion and dual maintenance | Acceptable for MVP; add a `_actual_source` field or comment before expanding to more types |
| Single `games` table for all item types | No schema complexity; all existing queries work | "games" is a misleading name; platform-centric queries produce noise for non-game types | Acceptable while collection is personal and small; becomes a problem if filtering/reporting by type becomes a major feature |
| Storing `cover_url` as a plain `String` | Simple to work with | No validation that the URL is local vs. external; easy to accidentally save an external URL | Never acceptable without the `cache_remote_cover()` save-path gate |
| Using `window.confirm()` for overwrite warnings in `fillFromIgdb` | Zero UI work | Poor mobile UX; blocks the JavaScript thread; cannot be styled | Acceptable while mobile use is secondary; replace with a modal when mobile polish is prioritized |
| `autogenerate`-based Alembic migrations without guards | Fast to generate | Will crash on live DBs that have already run partial migrations | Never acceptable for `ADD COLUMN` operations — always add `_column_exists()` guards |

## Integration Gotchas

| Integration | Common Mistake | Correct Approach |
|-------------|----------------|------------------|
| IGDB (Twitch OAuth) | Not refreshing the token before it expires; the in-memory cache `_igdb_token_cache` uses `expires_at` but only within a single process lifetime | Subtract a safety margin (already done: `-60` seconds) and ensure the cache is invalidated on process restart |
| IGDB cover URLs | Using the raw URL which starts with `//` (protocol-relative) | Prepend `https:` and replace `t_thumb` with `t_cover_big` (already done in `lookup_igdb_title`) — don't change this pattern |
| eBay Browse API | Hardcoded `EBAY_DE` marketplace ID — searches are German marketplace only | Fine for a personal DE-locale app; document it; do not assume it works for non-EU items without testing |
| UPCItemDB trial tier | Trial endpoint (`/prod/trial/lookup`) has very low rate limits (100 lookups/day) | The code already handles this by checking for `UPCITEMDB_API_KEY`; ensure the user is warned when trial limits are exhausted (HTTP 429) |
| ComicVine | API key passed as a query parameter, which gets logged in access logs | Acceptable for a personal tool; document it; do not log full request URLs if sensitive |
| GameTDB | No authentication, but returns cover URLs that are publicly hosted and stable | Use these URLs with confidence; caching is still recommended for offline/resilience |
| `cache_remote_cover()` | Does not retry on transient network failures; silently returns the original URL on any error | The fallback to the original URL is correct behavior; just ensure the caller checks whether the returned URL is still external |

## Performance Traps

| Trap | Symptoms | Prevention | When It Breaks |
|------|----------|------------|----------------|
| `cache_remote_cover()` called synchronously in the save path | Slow saves (up to 15s timeout per image) if the external host is slow | The function is `async`; ensure the save endpoint is also async and awaits it — it already is in the enrich endpoint | At first item with a slow CDN host |
| Bulk enrich iterates items sequentially with `await` per item | Bulk enrich for 20 items with 15s timeouts each = up to 5 minutes | The job already runs in a background task; progress is tracked via `jobs.update()`; do not move this to a synchronous endpoint | At 20+ items |
| Stats queries do not use indexes on `item_type` or `is_wishlist` | Slow stats page as collection grows | The `games` table has no index on `item_type`; add one if stats queries become slow | At 500+ items of mixed types |
| eBay search fires two HTTP requests when the first returns empty (`_do_search` fallback) | Double latency on unmatched searches | Acceptable; the fallback is bounded to one retry | Per search |

## Security Mistakes

| Mistake | Risk | Prevention |
|---------|------|------------|
| API keys stored in `app_meta` DB table (plaintext) | Anyone with DB read access can extract all API keys | Acceptable for a personal single-user tool; document the risk; do not expose the DB file over the network |
| `require_admin_access` dependency on bulk enrich but not on individual item save | Bulk enrich has a guard; individual saves do not — fine for a personal tool, but worth noting if ever exposed to a network | Confirm `ADMIN_ACCESS` env var is set before exposing the app on a LAN |
| Cover upload accepts any file, checks only size (5 MB) | Malicious file upload if app is exposed to the internet | Not a risk for a personal local app; if ever hosted publicly, add MIME type validation beyond just checking the Accept header |

## UX Pitfalls

| Pitfall | User Impact | Better Approach |
|---------|-------------|-----------------|
| Search provider dropdown defaults to "Games (IGDB/RAWG/GTDB)" when adding a figure or comic | User searches for "Nendoroid Rem" and gets zero game results; must manually switch provider | Auto-switch `searchProvider` based on selected `item_type` (e.g., `item_type === 'figure'` → default to 'mfc', `item_type === 'comic'` → 'comicvine') |
| `hobbydb` and `mfc` source labels in the result list are misleading — data actually comes from eBay auction listings | User selects what they think is authoritative HobbyDB data but gets a random eBay listing title | Add a "(via eBay)" note to these source badges, or show the eBay item ID so users understand provenance |
| All item types use the same completeness options ("CIB", "Box + Game", etc.) | "Game + Manual" completeness makes no sense for a Funko Pop | Filter completeness options by `item_type` in the form, or use a free-text field for non-game types |
| `item_type` is set to `'figure'` even when selecting a HobbyDB result, regardless of whether it's a Funko Pop | Funko Pops end up with `item_type = 'figure'`; the 'funko' type is never auto-set | Update `fillFromIgdb()` so `hobbydb` source sets `item_type = 'funko'` to match the dropdown option |
| Cover image in search results is loaded directly from external URLs (before caching) | Broken cover thumbnails in the search result list if the CDN blocks hotlinking | This is a search-time display issue only — acceptable — but add an `onerror` fallback on the `<img>` tag in the result list |

## "Looks Done But Isn't" Checklist

- [ ] **Image caching:** `cache_remote_cover()` called at save time in `create_game()` — verify `cover_url` in DB starts with `/uploads/` not `https://`
- [ ] **Image caching on update:** `update_game()` also passes the incoming `cover_url` through `cache_remote_cover()` — verify updating an item with a fresh external URL doesn't bypass caching
- [ ] **Migration idempotency:** New migrations have `_column_exists()` guards — verify by running `alembic upgrade head` twice in a row with no error
- [ ] **Migration chain continuity:** Each new migration's `down_revision` points to the correct previous revision ID — verify `alembic history --verbose` shows a linear chain with no broken links
- [ ] **eBay token failure visible to user:** When `get_ebay_token()` returns `None`, HobbyDB/MFC search returns `{"error": "eBay API credentials missing.", "results": []}` — verify the frontend shows this error rather than silently displaying an empty result list
- [ ] **Non-game platform_id nullable:** Saving a figure with no platform selected succeeds — verify `platform_id = None` does not raise a DB constraint error
- [ ] **Item type persisted correctly:** After selecting a ComicVine result, `item_type` is set to `'comic'` in the saved record — verify via `GET /api/games/{id}`
- [ ] **Stats by_type includes all types:** After adding a figure and a Funko Pop, the stats endpoint returns entries for those `item_type` values — verify `by_type` in the stats response

## Recovery Strategies

| Pitfall | Recovery Cost | Recovery Steps |
|---------|---------------|----------------|
| External URLs already in the DB | MEDIUM | Run the existing bulk enrich endpoint (`POST /api/enrich/all`) which calls `cache_remote_cover()` on items missing covers; for items that already have external cover URLs, a one-off script that iterates `games` where `cover_url LIKE 'http%'` and calls `cache_remote_cover()` |
| Migration crash on `duplicate column name` | LOW | The `b7c8d9e0f1a2` pattern is already in place; for a crashed migration, use `alembic stamp <rev_id>` to mark it as applied, then re-run with existence guards |
| Schema drift (column in model but not in DB) | LOW-MEDIUM | Write a new migration with `_column_exists()` guard; run `alembic upgrade head`; no data is lost |
| eBay token expired mid-session | LOW | Token refresh is automatic on next request; no user action needed; if the env vars are missing entirely, user must add them in Settings |
| Bulk enrich job hung | LOW | Job state is tracked in memory (`jobs.py`); restart the server process; no data is corrupted |
